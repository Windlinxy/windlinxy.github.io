# 集合

  • HashMap 和 ConcurrentHashMap 的区别？
    • concurrentHashMap 的 key 和 value 不能为 null—— 二义性问题
    • HashMap 不支持并发操作，没有同步方法；ConcurrentHashMap 支持并发操作，通过 CAS 和 synchronized 来进行加锁
    • HashMap：1.8 之前数组 + 链表，之后数组 + 链表 + 红黑树（链表长度 > 10 且元素个数 > 64）
    • ConcurrentHashMap:1.8 之前 segment 数组 + 数组 + 链表 之后数组 + 链表 + 红黑树
  • HashMap 的数据结构；
    • HashMap 里面是一个数组，然后数组中每个元素是一个单向链表节点或红黑树节点
    • jdk1.7 采用头插法，扩容会逆序和环形链表
    • jdk1.8 加入了红黑树，采用了尾插法，避免出现逆序和死循环问题

  • put 方法的细节；
    • 在 put 时调用 hashcode 获得 hash 值
    • 然后判定 hash 表的长度，如果为 0 就 resize () 先初始化数组
    • 然后通过 & 运算与数组长度 - 1 相与定位到 key 的插入位置
    • 如果该位置为 null，就直接添加，有就判断两个 key 是否相同（先判断 hash，再 == 和 equals）
    • 如果两个 key 相等，进行 value 的覆盖，返回新值
    • 如果不相等就要考虑插入链表和红黑树，会先判断头结点类型，如果为树节点调用 putTreeVal
      • 如果是链表节点，会挨个比较，相等就覆盖旧值
      • 都不相等最后就插入链表尾端，再去判断 binCount 也就是遍历节点数目是不是大于等于 7，会调用 treeifyBin 将链表转换成红黑树
  • HashMap 为什么线程不安全？
    • 数据覆盖问题，两个线程在 put 的时候对 map 中 entry 数组同一个位置进行添加元素，都完成了检测 hash 碰撞，导致数据会覆盖
  • ConcurrentHashMap 是如何保证线程安全的？
    • cas 和 synchronized
  • CourrentHashMap 1.8 比 1.7 的优化在哪里？
    • 1.7 采取 segment 分段锁的形式，采用在访问时对一段数据加锁，1.8 的锁颗粒度更小，是对数组每个元素加锁，访问时只锁单节点，也就是树或是链表的头部
  • CourrentHashMap 对 HashMap 的优化？
    • 保证了并发下的安全性，扩容时可以并发地进行扩容，当线程执行 put 操作会判断是否正在扩容，并在 put 的同时协助进行扩容

  • 线程有哪些状态？

    • NEW RUNNABLE BLOCKED WAITING TIME_WAITING TERMINSTED

  • 你对线程安全的理解是什么？

    • 当多个线程访问某个类时，不管运行时环境采用何种调度方式或者这些线程将如何交替执行，并且在调用代码中不需要任何额外的同步，这个类都能表现出正确的行为，那么这个类就是线程安全的。

  • sleep () 方法和 wait () 方法区别？

    • wait 是对象的方法，必须在 synchronized 中执行，是对象的方法，需要 notify 或 notifyAll 来唤醒（带时间的可以自主唤醒），会释放锁
    • sleep 是线程的方法，多用于线程休眠，wait 则用于多线程之间的同步通信，会让出 cpu

  • 偏向锁是什么？轻量级锁是什么？

    • 偏向锁是一种针对单线程执行时使用的优化手段。
    • 轻量级锁是多个线程交替持有锁但是没有竞争，只要简单的进行 CAS 操作即可避免线程上下文切换等开销

  • 讲一讲 synchronized 锁升级过程？

    • 线程第一次获取的时候会通过 CAS 操作将对象头的锁标志位变成偏向锁，并将对象头中的 ThreadID 改成自己的 ID。
    • 当前线程使用 CAS 将对象头的 mark Word 锁标记位替换为锁记录指针，如果成功，当前线程获得锁。
      • 如果失败，表示其他线程竞争锁，当前线程尝试通过自旋获取锁
      • 如果自旋失败，升级为重量级锁

  • CAS 了解多少？

    • 比较并替换，CAS 正是乐观锁的核心算法实现。

  • Java 的 JUC 包里有没有现成的类可以解决 CAS 的 ABA 问题？

    • AtomicStampedReference
    • ABA 问题是指在 CAS 操作时，其他线程将变量值 A 改为了 B，但是又被改回了 A，等到本线程使用期望值 A 与当前变量进行比较时，发现变量 A 没有变，于是 CAS 就将 A 值进行了交换操作，但是实际上该值已经被其他线程改变过，这与乐观锁的设计思想不符合。

  • AQS 了解多少？

    • AQS 是一个用来构建锁和同步器的框架
    • 核心思想：如果被请求的共享资源空闲，就将当前线程设置为工作线程，并且将共享变量设置为锁定状态。如果被请求的共享资源占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制（该机制由双向队列实现，将暂时获取不到锁的线程加入到队列中）。

  • 向线程池中提交任务的过程？

    • 如果正在运行的线程数量小于 corePoolSize，那么马上创建线程运行这个任务
    • 如果正在运行的线程数量大于或等于 corePoolSize，那么将这个任务 放入队列
    • 如果这个时候队列满了且正在运行的线程数量还小于 maximumPoolSize，那么还是要创建非核心线程立刻运行这个任务
    • 如果队列满了且正在运行的线程数量大于或等于 maximumPoolSize，那么线程池会 启动饱和拒绝策略来执行

    • DiscardOldestPolicy，将最早进入队列的任务删除，之后再尝试加入队列

  • 请说一下 ThreadLocal 的作用和使用场景？

    • 在 Java 的多线程并发执行过程中，为了保证多个线程对变量的安全访问，可以将变量放到 ThreadLocal 类型的对象中，使变量在每个线程中都有独立值，不会出现一个线程读取变量时被另一个线程修改的现象。
    • ThreadLocal 的主要价值在于线程隔离，ThreadLocal 中的数据只属于当前线程，其本地值对别的线程是不可见的，在多线程环境下，可以防止自己的变量被其他线程篡改。另外，由于各个线程之间的数据相互隔离，避免了同步加锁带来的性能损失，大大提升了并发性的性能。
    • 可以为每个线程绑定一个用户会话信息、数据库连接、HTTP 请求等，这样一个线程所有调用到的处理函数都可以非常方便地访问这些资源。常见的 ThreadLocal 使用场景为数据库连接独享、Session 数据管理等。
    • 可以为每个线程绑定一个数据库连接，使得这个数据库连接为线程所独享，从而避免数据库连接被混用而导致操作异常问题。（hibernate）

# Redis

  • Redis 的 key 有什么特点？涉及的时候有什么要注意的吗？

    • 以业务名为 key 前缀，用冒号隔开，以防止 key 冲突覆盖。如，live:rank:1
    • 确保 key 的语义清晰的情况下，key 的长度尽量小于 30 个字符。
    • key 禁止包含特殊字符，如空格、换行、单双引号以及其他转义字符。
    • Redis 的 key 尽量设置 ttl，以保证不使用的 Key 能被及时清理或淘汰。

  • Redis 的 value 有哪些类型？

    • String、list（quicklist）、hash（ziplist、ht）、set(ht、intset)、zset(skiplist、ziplist、（ht）)

  • Redis 过期键的删除策略有哪些？

    • 惰性删除 - 在键过期时，放任不管，如果后来有请求来查询该键，检测到该键已经过期了，此时才将这个键删除掉。
      • 节约 cpu 性能，到了必须删除的时候才执行删除。
      • 内存压力较大，会出现长期占用内存的数据。
    • 定期删除 - 每隔一段时间来删除过期键
    • 定时删除 - 在设置键的过期时间的同时，设置一个定时器，当键过期了，定时器马上把该键删除。
      • 节省内存，到时间就删除，快速释放不必要的内存占用。
      • CPU 压力较大，无论 CPU 此时负载情况如何，均占用 CPU 来执行删除，会影响 Redis 服务器的响应时间和指令吞吐量。

  • redis 内存淘汰策略 易丢失数据（可能会过期的数据集，server.db [*].expire）

    volatile-lru：挑选最近最少使用的数据淘汰

volatile-lfu：挑选最佳使用次数最少的数据淘汰 volatile-ttl：挑选将要过期的数据进行淘汰 volatile-random：随机选择数据淘汰 检测全库数据 (所有数据集，server.db [*].dict）

alikeys-lru：挑选最近最少使用的数据淘汰

alikeys-lfu：挑选最佳使用次数最少的数据淘汰 alikeys-random：随机选择数据淘汰 放弃数据逐出

no-enviction：禁止数据逐出，会引发OOM(out of memory)

  • aop 实现原理

    • 在编译时， Spring 会使用 AspectJ 编译器将切面代码编译成字节码文件。在运行时， Spring 会使用 Java 动态代理或 CGLIB 代理生成代理类，这些代理类会在目标对象方法执行前后插入切面代码，从而实现 AOP 的功能。
    • Spring AOP 可以使用两种代理方式：JDK 动态代理和 CGLIB 代理。如果目标对象实现了至少一个接口，则使用 JDK 动态代理；否则，使用 CGLIB 代理。下面分别介绍这两种代理方式的实现原理。
      • JDK 动态代理会在运行时生成一个实现了目标对象接口的代理类，该代理类会在目标对象方法执行前后插入切面代码。
      • CGLIB 代理是一个基于字节码操作的代理方式，它可以为没有实现接口的类创建代理对象。CGLIB 代理会在运行时生成一个目标对象的子类，并覆盖其中的方法，以实现 AOP 的功能。

  • spring 自动装配？

    • 自动装配就是自动地把第三方依赖的 jar 包中的类装入到 IOC 容器中，它依赖于启动类的注解 SpringBootApplication 注解中 EnableAutoConfiguration 注解
    • @AutoConfigurationPackage：主配置类（@SpringBootApplication 标注的类）的所在包以及下面所有子包里面的所有组件扫描到 Spring 容器。
    • @Import 注解，这个注解中引入了一个 AutoConfigurationImportSelector 类，该类实现 ImportSelectors 方法来动态的加载第三方 jar 包中 META-INFO 目录下 spring.factories 文件中符合条件的配置类，实现动态的条件装配（@Condtion 约定大于配置）

    • 数据校验：TCP 报文头有校验和，用于校验报文是否损坏
    • 超时重传：如果发送一直收不到应答，可能是发送数据丢失，也可能是应答丢失，发送方再等待一段时间之后都会进行重传。
    • 序列号控制：保证数据是按序、完整到达
    • 流量控制：当接收方来不及处理发送方的数据，能通过滑动窗口，提示发送方降低发送的速率，防止包丢失。
    • 拥塞控制：网络层拥堵造成的拥塞，包括慢启动，拥塞避免，快速重传三种机制
    • 合理分片：tcp 会按最大传输单元 (MTU) 合理分片，接收方会缓存未按序到达的数据，重新 排序后交给应用层。
